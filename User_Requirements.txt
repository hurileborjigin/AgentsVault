Agent Vault — Architecture & Engineering Specification

1. Purpose

Agent Vault is an open-source CLI-first retrieval system designed for other coding agents and developer tools to ask grounded questions about a specific project or knowledge domain.

The system supports two primary workflows:

Ingestion: recursively parse local project documents and assets, transform them into searchable representations, and store them in a Supabase-backed vector index.

One-shot query: answer a single question from the indexed knowledge base, return a grounded response with citations, export the exchange as markdown, and terminate.

This is not a chat application. It is a deterministic, auditable, agent-facing CLI tool.

2. Product Goals

Primary goals

Provide a clean CLI interface that other agents can invoke programmatically.

Support ingestion of mixed project knowledge sources.

Retrieve relevant context and generate grounded answers with references.

Persist every query result as a markdown artifact.

Use strong software design boundaries so storage, models, and parsers are swappable.

Non-goals for v1

Continuous conversations

Web UI

Multi-user collaborative platform

Background daemon

Real-time sync or file watching

Fully general multimodal reasoning engine

4. Primary CLI Commands

4.1 Ingest

agent-vault ingest --source <path>

Example:

agent-vault ingest --source ./docs --project payment-service

Behavior:

recursively scan the source directory

detect supported files

parse and normalize content

generate searchable chunks

compute embeddings

store metadata and vectors in Supabase

Useful future flags:

agent-vault ingest --source ./docs --project payment-service --reindex
agent-vault ingest --source ./docs --project payment-service --dry-run
agent-vault ingest --source ./docs --project payment-service --include-images

4.2 Ask

agent-vault ask "How does the retry logic work?"

Example:

agent-vault ask "How does the ingestion pipeline handle diagrams?" --project payment-service

Behavior:

embed the query

retrieve relevant chunks from Supabase

assemble grounded context

generate a one-shot cited answer

export the result as a markdown file

stop after the answer is produced

Useful future flags:

agent-vault ask "How does auth work?" --project payment-service --top-k 8
agent-vault ask "How does auth work?" --project payment-service --out ./.conversations
agent-vault ask "How does auth work?" --project payment-service --format md

4.3 Configure

agent-vault configure

Behavior:

The tool needs a stable and explicit way to know which provider and model to use. A dedicated configure command makes provider selection discoverable, audit-friendly, and consistent across commands.

10. Ingestion Pipeline

The ingestion flow must be implemented as a stage-based pipeline.

Stage 1: Source discovery

Input:

--source <directory>

Behavior:

recursively scan directory

collect supported files only

normalize file paths

compute checksums

create a manifest

Stage 2: Parsing and extraction

Text and markdown

read raw content

normalize whitespace

preserve headings where possible

PDF

extract page text

preserve page number metadata

optionally extract embedded image context later

Image

For v1, use a text-first multimodal strategy:

run OCR when text is present

generate a brief visual caption or description

store both as searchable text assets

This avoids premature complexity while still making diagrams and screenshots retrievable.

Stage 3: Chunking

Recommended strategy:

structure-aware chunking when headings are available

fallback token-window chunking otherwise

Suggested defaults:

target chunk size: 600 to 1000 tokens

overlap: 80 to 150 tokens

Metadata per chunk should include:

project id

source path

file type

page number when applicable

section heading when available

modality

chunk index

checksum

ingestion timestamp

Stage 4: Embedding

pass chunk text to the configured embedding provider

validate vector dimension consistency

persist vectors and metadata in Supabase

Stage 5: Persistence

upsert document entries

replace or upsert chunk rows based on document checksum

write ingestion job summary

11. Multimodal Design Strategy

v1 recommendation

Treat multimodal retrieval as searchable text representations of non-text content.

For images and image-heavy PDFs:

OCR result becomes ocr-text

visual caption becomes image-caption

Benefits:

simpler architecture

one retrieval pipeline

easier debugging

cleaner citations

better interoperability for agent use

v2 extension path

Later versions may add:

dedicated image embeddings

hybrid text plus image retrieval

figure-region extraction from PDFs

diagram-aware parsing

12. Query Pipeline

Step 1: receive question

Input:

agent-vault ask "specific question about the domain"

Step 2: load configuration

read provider and model from local config

validate env vars for chosen provider

Step 3: embed query

use configured embedding model

Step 4: retrieve candidate chunks

vector search top 20 to 40

optionally filter by project or file type

Step 5: context reduction

apply score threshold

select best 5 to 8 chunks

preserve metadata for citations

Step 6: answer generation

build grounded prompt

instruct model to answer only from retrieved context

require citations or explicit uncertainty

Step 7: export markdown log

create daily folder if missing

create a unique markdown file for the query

save question, answer, citations, and retrieved evidence

Step 8: print terminal summary

concise answer summary

saved file path

13. Retrieval Best Practices

Recommended retrieval strategy for v1

vector search for breadth

metadata filtering for isolation

strict context budget for answering

Strong recommendation

Do not use only top-3 raw vector results. Retrieve more broadly first, then reduce.

Suggested flow:

top 20 to 40 initial retrieval

apply project and modality filters

select top 5 to 8 final chunks

answer from final context only

Project namespace

A project namespace must be supported to avoid mixing unrelated knowledge bases.

Examples:

agent-vault ingest --source ./docs --project payments-api
agent-vault ask "How does retry logic work?" --project payments-api

If no explicit project is supplied, the default may be inferred from the current working directory name.

14. Supabase Data Model

14.1 documents

create table documents (
  id uuid primary key default gen_random_uuid(),
  project_id text not null,
  path text not null,
  file_type text not null,
  checksum text not null,
  title text,
  metadata jsonb not null default '{}'::jsonb,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  unique(project_id, path, checksum)
);

14.2 chunks

create table chunks (
  id uuid primary key default gen_random_uuid(),
  document_id uuid not null references documents(id) on delete cascade,
  chunk_index int not null,
  modality text not null,
  content text not null,
  embedding vector(1536),
  embedding_model text not null,
  metadata jsonb not null default '{}'::jsonb,
  created_at timestamptz not null default now()
);

14.3 ingestion_jobs

create table ingestion_jobs (
  id uuid primary key default gen_random_uuid(),
  project_id text not null,
  source_path text not null,
  status text not null,
  stats jsonb not null default '{}'::jsonb,
  started_at timestamptz not null default now(),
  finished_at timestamptz
);

Recommended indexes

btree on documents(project_id, path)

gin on documents(metadata)

gin on chunks(metadata)

vector index on chunks.embedding

15. Local Markdown Log Design

Folder rule

exactly one folder per date

multiple markdown files allowed inside that folder

Recommended structure:

.conversations/
└─ 2026-03-12/
   ├─ 2026-03-12T10-15-00_retry-logic.md
   ├─ 2026-03-12T11-03-44_auth-flow.md
   └─ 2026-03-12T16-48-29_pdf-diagrams.md

Naming rule

Use:

timestamp

slugified question prefix

optional collision suffix

Markdown format

---
id: 9c2e...
created_at: 2026-03-12T16:48:29+01:00
question: "How does the ingestion pipeline handle diagrams?"
provider: "openai"
model: "gpt-4.1"
embedding_model: "text-embedding-3-large"
project: "payment-service"
top_k: 8
sources:
  - docs/architecture.pdf
  - docs/ingestion.md
---

# Question
How does the ingestion pipeline handle diagrams?

# Answer
...

# Citations
1. docs/architecture.pdf, page 3
2. docs/ingestion.md, chunk 5

# Retrieved Context
## Source 1
...

This format makes logs inspectable, reproducible, and agent-friendly.

Behavioral rules:

one bad file must not abort the whole ingest job unless failure threshold is exceeded

errors must be surfaced clearly in CLI output

doctor should explain how to repair common setup issues

stack traces should only appear in debug mode

20. Idempotency and Re-indexing

Ingestion must be idempotent.

Recommended strategy:

compute file checksum before parsing

compare against existing stored checksum

skip unchanged files by default

support explicit --reindex to force refresh

This prevents duplicate chunks and reduces embedding cost.

23. Security and Operational Considerations

never commit API keys

keep provider credentials in environment variables

store only non-secret model selection in local config

avoid storing raw full documents in logs unless intentionally needed

prefer service-role usage only where operationally justified

ensure markdown exports do not accidentally leak sensitive environment values

26. Final Architectural Position

The best architecture for Agent Vault is:

CLI-first

clean architecture / ports-and-adapters

pipeline-oriented ingestion

text-first multimodal retrieval in v1

Supabase-backed vector storage

local markdown audit logs

interactive provider/model configuration via keyboard

This gives you a system that is practical to build, easy to reason about, friendly to contributors, and well suited for programmatic use by external agents.
